# AI Agent에게 구조를 주면, 비싼 모델이 필요 없다 — PEDAL 워크플로

![PEDAL Cycle](../public/PEDAL.png)

AI Agent에게 기능 하나를 통째로 맡겨본 적 있으신가요?

"로그인 기능 만들어줘"라고 하면, Agent는 열심히 코드를 써내려갑니다. 그런데 결과를 보면 — 설계가 없고, 에러 처리가 빠져 있고, 보안은 고려도 안 되어 있습니다. 왜? Agent에게 **"잘 만들어"** 라고만 했지, **"어떤 과정을 거쳐 만들어"** 라고는 안 했기 때문입니다.

**PEDAL**은 이 문제에 대한 답입니다.

---

## PEDAL이란

PEDAL은 전통적인 품질 관리 방법론 **PDCA(Plan-Do-Check-Act)** 사이클을 AI Agent 기반 소프트웨어 개발에 맞게 재설계한 워크플로입니다.

| 단계 | 의미 | 하는 일 |
| --- | --- | --- |
| **P** | Plan | 목표, 범위, 요구사항, 리스크를 문서로 정리 |
| **E** | Engineering | 아키텍처, API 설계, 데이터 모델을 명세 |
| **D** | Do | Engineering 문서를 따라 구현 |
| **A** | Analyze | 설계 vs 구현을 비교하여 갭 분석 |
| **L** | Learn | Wiki로 지식 축적, 완료 보고서 작성 |

여기에 **Iterate**(Analyze-Do 사이 자동 반복)와 **Archive**(산출물 보관)가 더해집니다.

각 문서 생성 단계마다 **크로스 리뷰**가 끼어들고, Analyze에서 품질 기준을 충족하지 못하면 **자동으로 Iterate**합니다. 이 구조 덕분에, Agent의 "한 방" 성능에 의존하지 않고도 일관된 품질을 보장할 수 있습니다.

```
[Plan] → [Review] → [Engineering] → [Review] → [Do] → [Analyze] → [Review] → [Learn] → [Review] → [Archive]
                                                           │    ↑
                                                           ↓    │
                                                        [Iterate]
```

## 핵심 1: 크로스 리뷰 — 에코 챔버를 깨는 구조적 장치

### 같은 AI끼리는 같은 실수를 한다

GPT에게 코드를 작성하게 하고, GPT에게 리뷰를 시키면 어떤 일이 벌어질까요? 높은 확률로 "잘 작성되었습니다"라는 답이 돌아옵니다. 같은 모델은 같은 학습 데이터, 같은 추론 패턴을 공유합니다. 한쪽이 간과한 부분을 다른 쪽도 간과할 확률이 높습니다.

이것은 코드 리뷰에서 자기 코드를 자기가 리뷰하는 것과 같습니다. 아무리 꼼꼼해도 자신의 사고 패턴 안에 갇히게 됩니다.

### 다른 회사의 AI가 리뷰한다

PEDAL의 크로스 리뷰는 **반드시 다른 회사의 AI 모델**이 수행합니다.

| 작성 Agent | 리뷰 Agent |
| --- | --- |
| Cursor (Claude) | Gemini CLI |
| Gemini CLI | Cursor (Claude) |

서로 다른 회사의 모델은 학습 데이터도, 정렬(alignment) 방식도, 강점과 약점도 다릅니다. Claude가 놓치는 보안 이슈를 Gemini가 잡아내고, Gemini가 과도하게 복잡하게 설계한 부분을 Claude가 단순화 제안을 합니다.

이것은 인간 팀에서 "다른 팀 출신 개발자의 리뷰"가 더 통찰력 있는 것과 같은 원리입니다.

### 맹목적 수용 금지

크로스 리뷰에서 한 가지 더 중요한 규칙이 있습니다: **리뷰 결과를 맹목적으로 수용하지 않는다.**

리뷰 Agent도 틀릴 수 있습니다. 작성 Agent는 리뷰의 각 항목을 비판적으로 평가하여:
- **유효한 지적**은 수용하고 문서를 업데이트
- **부정확한 지적**은 근거를 들어 거절

이 과정이 없으면, 리뷰 Agent의 hallucination이나 맥락 부족에 의한 잘못된 지적이 그대로 반영되는 위험이 생깁니다.

```
리뷰 결과 수신
    │
    ├── Critical: 하드코딩된 시크릿 발견
    │   └── ✅ 수용 → env 변수로 이동
    │
    ├── Warning: Redis 대신 Memcached 권장
    │   └── ❌ 거절 → 프로젝트에 이미 Redis 인프라 존재, 전환 비용 대비 이점 없음
    │
    └── Info: 함수명을 더 명시적으로 변경 제안
        └── ✅ 수용 → 리네이밍 적용
```

이 "비판적 수용" 메커니즘은 크로스 리뷰의 이점을 살리면서도, 맥락을 가장 잘 아는 작성 Agent가 최종 결정권을 갖도록 합니다.

## 핵심 2: Learn 단계의 Agent 지식화

### Agent는 기억하지 못한다

AI Agent의 가장 큰 한계는 **세션 간 기억이 없다**는 것입니다. 오늘 3시간에 걸쳐 로그인 기능을 구현하면서 프로젝트 구조를 완벽히 파악했더라도, 내일 결제 기능을 시작하면 다시 처음부터 파악해야 합니다.

매번 Agent에게 긴 컨텍스트를 주입하거나, Agent가 스스로 코드베이스를 탐색하느라 시간과 토큰을 소비합니다. 그리고 탐색 과정에서 놓치는 부분이 생기면, 그것이 곧 버그로 이어집니다.

### Wiki가 Agent의 장기 기억이 된다

PEDAL의 Learn 단계에서는 **Wiki 문서**를 생성합니다. 이 Wiki는 단순한 개발자용 문서가 아니라, **Agent가 다음 작업에서 참조할 수 있는 구조화된 지식**입니다.

Wiki에는 다음이 기록됩니다:
- 기능의 아키텍처와 설계 결정 이유
- 핵심 구현 세부사항과 파일 위치
- UI가 있는 기능의 경우 ASCII Art로 표현된 화면 레이아웃
- API 명세와 설정 방법
- 모듈 구조와 의존성

```
docs/wiki/user-auth.wiki.md
━━━━━━━━━━━━━━━━━━━━━━━━━━
# user-auth

## 아키텍처
- JWT 기반 인증, Refresh Token은 HttpOnly Cookie
- 미들웨어: src/middleware/auth.ts
- ...

## 파일 구조
src/features/user-auth/
├── components/LoginForm.tsx
├── api/auth.ts
├── hooks/useAuth.ts
└── types/auth.types.ts

## API
POST /api/auth/login   → { token, refreshToken }
POST /api/auth/refresh → { token }
DELETE /api/auth/logout → 204

## 화면 레이아웃
┌─────────────────────────┐
│       로그인 화면         │
├─────────────────────────┤
│  [이메일 입력]            │
│  [비밀번호 입력]          │
│  [로그인 버튼]            │
│  비밀번호 찾기 | 회원가입   │
└─────────────────────────┘
```

다음 사이클에서 Agent가 결제 기능을 개발할 때, `user-auth.wiki.md`를 참조하면 인증 미들웨어의 위치, API 명세, 토큰 전략을 즉시 파악할 수 있습니다. "파일을 하나하나 열어보면서 파악"하는 시간이 사라집니다.

### 지식이 쌓일수록 Agent가 똑똑해진다

PEDAL 사이클을 반복할수록 Wiki가 풍부해집니다. 프로젝트가 성숙할수록 Agent가 활용할 수 있는 컨텍스트가 넓어지고, 그만큼 더 정확한 판단을 내릴 수 있습니다.

```
1번째 사이클: Wiki 0개  → Agent가 모든 것을 탐색해야 함
3번째 사이클: Wiki 3개  → 핵심 기능의 구조를 즉시 파악
10번째 사이클: Wiki 10개 → 프로젝트 전체를 이해한 상태에서 작업
```

이것은 새 팀원이 프로젝트에 합류했을 때, 내부 문서가 잘 정리되어 있으면 온보딩이 빨라지는 것과 같습니다. PEDAL의 Wiki는 **Agent를 위한 온보딩 문서**입니다.

## 핵심 3: 체급이 낮은 Agent로도 충분하다

### "더 좋은 모델"이 답이 아닌 이유

AI 개발에서 가장 흔한 함정은 **"결과가 안 좋으면 더 비싼 모델을 써야지"** 라는 사고입니다.

물론 모델의 역량은 중요합니다. 하지만 프로세스 없이 최상위 모델을 쓰는 것보다, 구조화된 프로세스 위에 중급 모델을 올리는 것이 더 나은 결과를 내는 경우가 많습니다.

왜 그런지 분해해 보겠습니다.

### 구조가 역량을 보완하는 메커니즘

**1단계: Plan + Engineering이 가이드라인을 제공**

Agent가 "로그인 기능 만들어"라는 모호한 지시만 받으면, 스스로 요구사항을 추론하고, 설계를 결정하고, 구현해야 합니다. 이 모든 것을 한 번에 잘 할 수 있는 Agent는 최상위 모델뿐입니다.

하지만 Plan 문서에 "JWT 인증, Refresh Token은 HttpOnly Cookie, OWASP Top 10 준수"가 명시되어 있고, Engineering 문서에 API 엔드포인트와 데이터 모델이 정의되어 있으면? 중급 모델도 **"무엇을 만들어야 하는지"** 가 명확하므로 구현에 집중할 수 있습니다.

**2단계: 크로스 리뷰가 안전망 역할**

중급 모델이 실수를 할 확률은 더 높습니다. 하지만 크로스 리뷰가 이 실수를 잡아냅니다. 그리고 리뷰 Agent 역시 비싼 모델일 필요가 없습니다 — PEDAL의 크로스 리뷰에서 Gemini Flash같은 경량 모델을 리뷰어로 사용해도 효과적입니다. 핵심은 **"다른 관점"** 이지, **"더 뛰어난 관점"** 이 아닙니다.

**3단계: Iterate가 품질을 수렴시킨다**

한 번에 완벽한 결과를 내는 것은 어떤 모델에게든 어렵습니다. PEDAL은 이것을 요구하지 않습니다. 대신, Analyze에서 갭을 정량적으로 측정하고, 기준(match rate 90%, Critical 이슈 0건)을 충족할 때까지 자동으로 반복합니다.

```
Iteration 1: match rate 72% → 3개 이슈 수정
Iteration 2: match rate 85% → 2개 이슈 수정
Iteration 3: match rate 93% → 기준 충족, 완료
```

최상위 모델은 1회만에 90%를 넘길 수도 있지만, 중급 모델도 3회 반복이면 같은 수준에 도달합니다. 비용은 전자가 훨씬 높습니다.

**4단계: Wiki가 컨텍스트를 보강**

중급 모델의 또 다른 약점은 컨텍스트 활용 능력입니다. 하지만 Wiki에 프로젝트 지식이 구조화되어 있으면, 모델이 스스로 추론해야 할 부분이 줄어듭니다.

### 비용 대비 효과

```
┌──────────────────────────────────────────────────────┐
│  접근 방식 A: 최상위 모델 + 프로세스 없음              │
│  ──────────────────────────────────────              │
│  비용: $$$$$                                         │
│  결과: 들쭉날쭉 (한 번에 잘 되기도, 크게 빗나가기도)     │
│  재현성: 낮음                                         │
├──────────────────────────────────────────────────────┤
│  접근 방식 B: 중급 모델 + PEDAL                       │
│  ──────────────────────────────────────              │
│  비용: $$                                            │
│  결과: 일관적 (Iterate가 수렴을 보장)                   │
│  재현성: 높음                                         │
└──────────────────────────────────────────────────────┘
```

**결론**: 모델을 업그레이드하기 전에, 프로세스를 업그레이드하세요.

## Severity-Weighted Scoring: 정량적 품질 게이트

PEDAL의 Analyze 단계는 감이 아닌 숫자로 판단합니다.

| 심각도 | 가중치 | 조건 |
| --- | :---: | --- |
| Critical | x3 | **무조건 Iterate** (match rate 무관) |
| Warning | x2 | match rate < 90%일 때 |
| Info | x1 | match rate < 90%일 때 |

```
weightedScore      = Σ(심각도 가중치 × 이슈 수)
maxPossibleScore   = 전체 점검 항목 × 3
weightedMatchRate  = (1 - weightedScore / maxPossibleScore) × 100%
```

Critical 이슈가 하나라도 남아 있으면, 전체 match rate가 95%여도 Iterate가 강제됩니다. 보안 취약점이나 핵심 요구사항 누락이 수치에 묻히지 않도록 하는 장치입니다.

## 사람의 개입 지점: Human Gate

AI가 전부 알아서 하는 것은 아닙니다. PEDAL은 두 곳에 명시적 승인 게이트를 둡니다.

```
Plan → Engineering → ⛔ GATE 1 → Do → Analyze → ⛔ GATE 2 → Learn
```

- **Gate 1 (Engineering 후)**: 설계를 기반으로 구현하기 전, 사람이 설계를 검토하고 승인합니다. 잘못된 설계 위에 쌓인 구현은 되돌리기 어렵기 때문입니다.
- **Gate 2 (Analyze 후)**: 분석 결과를 확인하고, 범위 재조정이 필요한지 판단합니다.

Agent에게 자율성을 주되, 핵심 의사결정은 사람이 합니다.

---

## 마무리

PEDAL은 세 가지 관찰에서 시작했습니다:

1. **같은 AI끼리는 같은 실수를 반복한다** → 크로스 리뷰
2. **Agent는 세션이 끝나면 모든 것을 잊는다** → Learn 단계의 Wiki 지식화
3. **비싼 모델이 항상 답은 아니다** → 구조와 반복이 모델의 체급을 보상

자전거의 페달처럼, 한 바퀴 한 바퀴 돌릴 때마다 앞으로 나아갑니다. 중요한 것은 페달을 밟는 힘(모델의 성능)이 아니라, **기어와 체인(프로세스)이 제대로 연결되어 있는지**입니다.

PEDAL을 프로젝트에 도입하면, AI Agent가 "가끔 잘 되는 도구"에서 "반복 가능한 개발 파트너"로 바뀝니다.
