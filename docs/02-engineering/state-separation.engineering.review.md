# state-separation Engineering Review

> **Date**: 2026-04-16
> **Document**: docs/02-engineering/state-separation.engineering.md

## Critical

- **하드코딩된 브랜치 의존성** (섹션 2.2, 단계 6): 설계서에 `git checkout master`가 명시되어 있습니다. 이는 메인 브랜치 이름이 반드시 `master`라고 가정하며, 사용자가 항상 해당 브랜치와 동기화하기를 원한다고 전제합니다. 현대적인 프로젝트에서는 `main`을 주로 사용하며, 여러 기능을 동시에 개발할 때는 특정 업스트림 브랜치와 동기화해야 할 수도 있습니다. 이 로직은 `git symbolic-ref refs/remotes/origin/HEAD` 등을 활용하여 일반화되어야 합니다.

- **원자적 쓰기 및 Git 충돌 문제**: 단계 6("Shared Write")에서 다른 브랜치로부터 파일을 체크아웃한 후 쓰기를 수행합니다. 이 과정에서 `.pedal-status.shared.json`에 대한 로컬의 커밋되지 않은 변경 사항이 유실되거나, 파일이 더티(dirty) 상태일 경우 체크아웃이 실패할 위험이 있습니다. 이 "동기화" 단계에서 발생할 수 있는 Git 수준의 충돌을 처리하는 구체적인 전략이 부족합니다.

## Warning

- **JSON 병합 로직의 구체성 부족**: 섹션 2.2 단계 5에서 "데이터 병합 및 업데이트"를 언급하고 있지만, `features` 객체 내에서의 충돌 해결 방식이 명시되지 않았습니다. 여러 워크트리에서 동시에 동일한 피처를 업데이트할 경우, 단순 덮어쓰기인지 아니면 딥 머지(deep merge)인지 정의가 필요합니다. `jq`와 같은 도구를 활용한 딥 머지 방식을 권장합니다.

- **락(Lock) 범위의 한계**: 섹션 4.2의 `Advisory Lock`은 로컬의 서로 다른 워크트리 간 경쟁 상태는 방지할 수 있지만, 다른 협업자가 원격에 푸시한 업데이트는 방어할 수 없습니다. 이 락이 '로컬 전용' 동시성 제어 메커니즘임을 문서에 명확히 명시해야 합니다.

## Info / Suggestions

- **전역 디렉토리 초기화**: `pedal-sync.sh` 스크립트에서 락 파일을 생성하기 전에 `mkdir -p ~/.pedal/${REPO_ID}`를 실행하는 로직을 명시적으로 포함해야 합니다.

- **결함 있는 락(Stale Lock) 처리의 견고함**: PID 확인 로직은 좋으나, OS에서 PID는 재사용될 수 있습니다. 1분의 타임아웃 임계값을 주된 판단 기준으로 삼는 것이 더 안전합니다.

## Verdict

**REVISE_REQUIRED**

이 설계서는 구조적으로 잘 정리되어 있고 핵심 요구사항을 잘 반영하고 있습니다. 하지만 하드코딩된 브랜치 참조와 "Git Sync" 병합 로직의 모호함은 다양한 저장소 설정에서 스크립트가 정상 작동하고 데이터 유실을 방지하기 위해 반드시 수정되어야 할 중요한 사항입니다.
